# 框架的测试目标

框架包含三大部分：
 * 负载模拟
 * 请求调度
 * 数据收集

## 框架的基本原理

本框架采用JMeter的设计思路，算是一个简易版本的JMeter：

建立一个线程池，多线程运行测试用例产生大量负载，在运行过程中，框架记录每次请求的时长，并发送相关事件到事件收集器，然后通过事件分析器分析数据，并展示到前台。

当前需求比较简单，就是记录每个请求的时长。

涉及到5个主要的模块：

* 负载构建（器）
* 请求调度（器）
* 事件收集（器）& 事件容器
* 事件分析（器）
* UI展示

UI展示模块采用前后端分离的方式，整个测试工具会内置一个Web服务器（Tomcat或者Jetty）。这个功能等有机会了再进行。目前，采用AB Tools (Apache Httpd的一个建议测试工具)的方式，执行完所有的测试用例后，直接输出测试的结果。

#### 相关概念

* 测试步骤

  在JMeter中，有个概念叫做“取样器”，表示执行一个动作。一个测试用例可能由多个“取样器”组成。

  在这个框架中，直接叫做“测试步骤”，直接对应被测试系统的一个API，直观明了。

  QA概念中的一个“测试用例”，由**一个或者多个**“测试步骤”组成。

  举个例子：一个测试用例就是测试一个API，也就是一个Http Request；一个测试用例也可能包含多个API，多次HTTP Request。

  一个“测试步骤”，会封装成Java的线程对象，并赋予一个名称，这个很重要，它用于跟踪和分析每个测试步骤的运行情况。

* 测试用例

  一个或者多个“测试步骤”组成，一个“测试用例”对应一个测试场景。

* 配置元件

  每个“测试用例”包含的多个部分，大致有如下几种类型： 

  * 请求使用的协议。例如：HTTP, HTTPS, TCP等等

  * 测试API的URL。URL又可以拆分成2部分：
    * Domain_Name  ( Host_Name/IP + Port_Number)，如：shopex.homecredit.cn
    
    * Request Path，如：/suning/phone/iPhone/11
    
      Domain_Name 与 Request Path组合成一个完整的请求URL，或者EndPoint。
    
  * API的请求方法，如：GET，POST，PUT，DELETE等等。如果是TCP，没有这个选项。暂时也不提供TCP的测试功能。

  * API的请求参数（GET：URL上的参数；POST/Update：Body），设置一段JSON。

    1. 如果是GET，框架会自动转成K-V的形式，放到URL上
    2. 如果是POST等请求类型，会完成的把JSON放到Body里传过去（Content-Type默认为application/json，其他格式的Body，以后再提供支持）。

* 前置处理器

  “前置处理器”也是JMeter中的概念，当JMeter的“取样器”发起请求前，可以用“前置处理器”做一些“准备”工作，比如参数化等等。我们沿用JMeter的这个概念，可用于“测试用例”中各个“测试步骤”之间传递数据使用。

* 后置处理器

  “后置处理器”也是JMeter中的概念，当JMeter的“取样器”收到响应后做一些工作，比如提取响应报文中的内容。我们沿用JMeter的这个概念，可用于“测试用例”中各个“测试步骤”之间传递数据使用。

* 定时器

  如果一个“测试用例”需要持续执行一段时间，那么“定时器”就是用来控制这件事情的。

* 控制器

  在JMeter中，控制器的作用，就是控制采样器的执行顺序。JMeter提供了很多控制器：循环控制器、IF控制器等等。

  在这个框架中，第一版本不提供控制器（哇哈哈，因为太复杂了）。

### 事件收集

每个TestCase运行前，框架会创建一个Event对象，记录TestCase运行前的Timestamp数据；TestCase结束后，在此记录当前的Timestamp数据，然后把此Event对象发送给事件收集器。

#### 事件

事件应该有多种。

* 对于测试API的事件，都应该具备如下基本的信息：

  `{`

  ​    "testCaseName": "获取用户的DutyDate",

  ​    “apiIdentify”:"用户登录",

  ​    "method":"GET",

  ​	"startTime": 98765492,

  ​    "endTime": 98876448

  `}`

这就是最基础的事件定义对象。如果一个特定事情发生没有特别的需求，就可以采用这个基础的事件对象；如果有特殊的需求，则从这个基础事件扩展。

* 对于平台信息的事件，应具备如下的基本信息：

  `{`

  ​    "cupUsage": "CPU的使用比例",

  ​    "memUsage": "内存使用比例",

  ​    "threads": "线程数（3组：正在使用线程数；空闲线程数；僵尸线程数）",

  ​    "timing":"采集数据时刻（格式：HH24:mm:ss）"
  
  `}`

#### 事件收集器

事件生层后，会被收集到内存中。这个收集的过程，类似一个Map-Reduce的过程，会根据事件的类型、测试用例等维度进行分类。这些分类，都是针对（性能、负载、压力）测试的需求而定的。

目前不提供定制的收集器，框架内部内置一些收集器，针对我们当前设定的几种测试需求。

## 几种测试的含义

 * 性能测试：
   <p>
   模拟用户负载来测试系统在负载情况下，系统的响应时间，吞吐量等。
   （这里的负载指的是用户并发数）所以它的目的是为了获取系统的指标。
   </p>
 * 负载测试：
   <p>
   在一定的软硬件环境上，通过不断的加大负载来确定在满足性能指标情况下
   所能够承受的最大用户数。所以它的目的是为了获取最大用户数。
   一般不超过80%cpu，正常情况工作下最大用户数数据。
   </p>

 * 压力测试
   <p>
   也叫强度测试。
   在一定的软件硬件环境下，通过高负载的手段来使服务器资源处于极限的状态，
   测试该系统在极限状态长时间运行是否稳定。包括系统指标，服务器性能指标。
   </p>
